# Roadmap

## Phase 1: MVP

目标：把抓包数据稳定地转成可分析、可局部回放的结构化资产。

- 定义 `CapturedRequest`、`CapturedResponse`、`NormalizedExchange`
- 完成标准化、脱敏、内容类型归一
- 实现 Replay Readiness 判定
- 支持手工挑选一段录制并串行回放
- 输出最小执行报告

验收标准：

- 同一接口的多次抓包能被归到同一模板
- 至少能区分“可回放请求”和“仅分析请求”
- 回放失败时能输出基础失败分类

## Phase 2: V1

目标：把录制流量转成真正可维护的自动化用例。

- 实现请求序列切片
- 实现依赖链识别与 step DAG 构建
- 实现变量提取候选评分与注入排序
- 实现默认断言生成与波动字段豁免
- 输出 `TestCase`、`ReplayReadyStep`、`AssertionRule`

验收标准：

- 一段录制流量能自动转成参数化用例
- 用例可在新运行上下文中重放
- 断言不会因为弱随机字段普遍误报

## Phase 3: V2

目标：让 AI 只在高价值、低确定性的节点参与。

- 接入多角色 Prompt 路由
- 增加 API 文档补全与语义命名
- 增加 AI Patch 生成、校验、合并
- 增加低置信升级人工确认
- 增加失败解释和修复建议

验收标准：

- deterministic 规则和 AI patch 能被清晰区分
- AI 输出不合法时系统可拒收
- 低置信 patch 不会直接污染主用例

## Phase 4: V3

目标：形成可扩展、可协作、可持续优化的工程化体系。

- 缺陷聚合与 defect key 稳定化
- Prompt 评估闭环
- 失败模式库沉淀
- 案例级改写建议与回放修复闭环
- 文档、Prompt、执行报告三者联动

验收标准：

- 相似失败可自动聚合
- Prompt 迭代有明确评分基线
- 外部团队可按实施手册独立复现能力
