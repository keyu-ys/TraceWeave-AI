# Prompt 工程包

## 目标

这里定义的不是零散 Prompt 片段，而是一套可被系统稳定调用的角色契约。每个角色都必须输出结构化结果，且遵守统一边界：

- 不依赖私有业务字段名
- 不假设固定鉴权方式
- 不透传敏感 Header、Token、Cookie
- 低置信输出必须显式标记 `待人工确认`
- 优先输出结构化 Markdown 或结构化对象，而不是自由散文

## 统一输出框架

每个角色都应该按下面八段输出：

1. 角色定位
2. 输入契约
3. 输出契约
4. 决策规则
5. 禁止事项
6. 失败处理
7. 示例输入输出
8. 评估标准

## Role 1: capture analyst

### 角色定位

理解一批抓包交换的组成、边界和异常，不做业务臆测。

### 输入契约

- `NormalizedExchange[]`
- 聚类摘要
- 异常检测信号

### 输出契约

- 抓包摘要
- 异常清单
- 值得进入回放链路的请求建议
- 不确定项列表

### 决策规则

- 先按证据说明异常，再给建议
- 没有 replay value 的请求必须说明原因

### 禁止事项

- 不创造不存在的依赖链
- 不把敏感信息抄回输出

### 失败处理

- 输入过少时只输出观察结论，不给回放建议

### 示例输入输出

- 输入：10 条标准化交换，2 条 `analysis_only`
- 输出：建议保留 6 条进入场景草案，2 条标 `待人工确认`

### 评估标准

- 是否能区分事实与推断
- 是否能指出不适合回放的原因

## Role 2: api doc curator

### 角色定位

补全文档快照、改进字段命名、补充接口说明。

### 输入契约

- cluster 摘要
- `ApiDocSnapshot` 草案
- 多次抓包证据引用

### 输出契约

- 文档 patch
- 字段命名建议
- 必填/选填判断

### 决策规则

- 多次证据优先于单次证据
- 没有稳定证据时只能给 `suggest`

### 禁止事项

- 不凭单个样本宣布字段必填
- 不把高随机字段命名为业务主键

### 失败处理

- 聚类不稳定时直接返回证据不足

### 示例输入输出

- 输入：同模板 20 次抓包，字段 `statusLabel` 出现率 100%
- 输出：将其标为稳定响应字段，并补一句用途说明

### 评估标准

- 命名是否更清晰
- 必填与选填是否有证据支持

## Role 3: testcase orchestrator

### 角色定位

围绕一个 `TestCase` 草案补隐式依赖、变量命名和场景标题。

### 输入契约

- `TestCase` 草案
- `ApiDocSnapshot`
- 决策点说明

### 输出契约

- `AiPatchProposal[]`
- patch 依据说明
- 低置信项列表

### 决策规则

- 优先补字段级 patch，不输出整对象
- 与 deterministic 结果冲突时，降为建议

### 禁止事项

- 不删除已有关键断言
- 不重排 step 顺序，除非有强证据

### 失败处理

- 冲突多时直接请求人工审阅

### 示例输入输出

- 输入：一个变量名为 `value_1`
- 输出：给出 `resource_id` 的命名 patch，并引用证据

### 评估标准

- patch 是否结构化
- 证据引用是否充分

## Role 4: assertion synthesizer

### 角色定位

补充弱语义断言和结构断言增强建议。

### 输入契约

- `ReplayReadyStep`
- 响应结构摘要
- 波动字段列表

### 输出契约

- `AssertionRule` patch
- 每条断言的强度说明

### 决策规则

- 先状态、后结构、再语义
- 高波动字段只能做模式或豁免

### 禁止事项

- 不把随机值写成精确断言
- 不建议删除所有弱语义断言

### 失败处理

- 无法稳定生成时，只输出候选断言并标记待确认

### 示例输入输出

- 输入：字段类型稳定、值波动
- 输出：生成类型断言，不生成精确值断言

### 评估标准

- 断言是否稳健
- 是否清楚区分强断言和弱断言

## Role 5: replay diagnostician

### 角色定位

解释执行失败，并给出归因和修复方向建议。

### 输入契约

- `ExecutionRun`
- `ExecutionLog[]`
- 历史 defect 摘要

### 输出契约

- 失败归因摘要
- 关键证据
- 修复建议
- 是否需要人工复核

### 决策规则

- 按网络、协议、依赖、结构、断言顺序归因
- 没有证据不要升级为高置信结论

### 禁止事项

- 不建议无限重试
- 不把单次偶发失败定义成稳定缺陷

### 失败处理

- 根因竞争时输出多假设，不做单结论

### 示例输入输出

- 输入：响应成功但结构字段缺失
- 输出：`schema_break`，附字段路径证据

### 评估标准

- 归因顺序是否合理
- 证据和结论是否匹配

## Role 6: defect triager

### 角色定位

将多次失败聚合成稳定 defect 视图，并给出优先级建议。

### 输入契约

- 多个 `ExecutionRun`
- `DefectRecord` 草案
- 缺陷聚合键和证据集

### 输出契约

- 聚合建议
- 是否合并 / 拆分
- 优先级说明

### 决策规则

- failure class 不同默认不合并
- 关键路径不同默认不合并

### 禁止事项

- 不用真实实例 ID 做聚合依据
- 不把信息不足的失败强行合并

### 失败处理

- 证据不足时返回 `keep_separate`

### 示例输入输出

- 输入：3 次相同模板路径、同类缺失字段失败
- 输出：建议合并为一个 defect record

### 评估标准

- 是否降低重复 defect 噪音
- 是否避免误聚合

## Prompt 失败模式库

| 失败模式 | 表现 | 防护 |
| --- | --- | --- |
| 幻觉变量 | 模型创造不存在的依赖变量 | 强制证据引用 |
| 过度参数化 | 把所有字段都改成变量 | 提取评分阈值 |
| 敏感头透传 | 输出真实敏感头值 | 敏感字段拒收 |
| 无依据补断言 | 凭感觉写业务断言 | 要求结构证据 |
| 路径模板错误 | 把随机段当固定路径 | 先走 canonicalization |

## Prompt 评估建议

- 结构合法率
- 证据引用完整率
- 高风险 patch 拒收率
- 人工复核命中率
- Patch 最终采纳率
